# Reporting Fixtures

These fixtures are synthetic reporting inputs for the root CI and
Integration summary renderers. They are hand-curated test data, not generated
run output. They are not copied from GitHub Actions artifacts.

The fixture trees intentionally stay small while preserving the report contracts
and parser branches that the extracted workflow heredocs consumed:

- `ci/root/` contains representative fast-CI JUnit, coverage, Bandit, ESLint,
  and npm-audit artifacts for the summary rows and parser paths that need
  artifact input. Some green status cells intentionally come from fixed workflow
  env values rather than a redundant artifact file.
- `integration/root/` contains representative Integration JUnit, timing, TRX,
  and baseline files for every summary row.
- `integration/root/all-results/results-testclient/summary.md` is the synthetic
  Test Client conformance artifact feeding the aggregator's
  `load_test_client_conformance_summary()`. Its banner, KPI strip, and
  top-level section headings must mirror the wire shape emitted by
  `scripts/reporting/conformance_summary.py` (Test Client renderer).
  When that renderer's wire shape changes, update this file AND
  `expected/integration_summary.md` in the same commit.
- `expected/*.md` snapshots are generated by running the extracted summary
  modules with the fixed environment values in
  `tests/reporting/test_summary_snapshots.py`.

The purpose is byte-stable regression coverage for the extraction boundary, not
to mirror a particular live run.

## Fixture Policy

- Do not commit live CI artifact dumps, run logs, downloaded `test-results/`
  trees, or run-specific values.
- Keep committed artifact-shaped inputs small, deterministic, and
  scenario-owned. If a future edge case can be expressed more clearly as an
  inline string or a tiny builder, prefer that over adding files.
- Each committed input must exercise a visible report row, warning path,
  parser branch, or documented scenario. Remove unused inputs.
- `tests/baselines/integration-test-counts.json` in this fixture tree is a
  synthetic baseline shaped like the real Integration baseline; its
  `captured_from_run` value must stay synthetic unless the whole scenario is
  deliberately re-captured from one coherent run.
- Golden snapshots under `expected/` are byte-sensitive. Update them only for
  an intentional report wording/layout change, and review the Markdown diff as
  user-facing report UX.

## Refresh Rules

Run `python -m pytest tests/reporting -q` after any reporting fixture or
snapshot change. A snapshot mismatch means either the report changed
intentionally and the expected Markdown needs review, or the fixture no longer
matches the extraction-parity scenario.
